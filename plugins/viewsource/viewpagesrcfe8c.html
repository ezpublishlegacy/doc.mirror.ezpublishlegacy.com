ViewManager & ContentViewProvider

The role of the ViewManager is to select the right template for displaying a given content or location. It aggregates objects called content view providers that implement eZ\Publish\MVC\View\ContentViewProvider interface.

Each time a content is to be displayed through the Content\ViewController, the ViewManager iterates over the registered ContentViewProvider objects and calls either getViewForContent() or getViewForLocation() depending on the context.

Provided ContentViewProvider implementations

NameUsage
ConfiguredContentViewProvider

Based on application configuration.
Formerly known as Template override system.

LegacyContentViewProvider

Forwards view selection to the legacy kernel by running the old content/view module.
Pagelayout used is the one configured in ezpublish_legacy.view.default_layout.

Custom ContentViewProvider

When to develop a custom ContentViewProvider

  • You want a custom template selection based on a very specific state of your application
  • You depend on external resources for view selection
  • You want to override the default one (based on configuration) for some reason

ContentViewProvider objects need to be properly registered in the service container with the ezpublish.content_view_provider service tag.

parameters:
    ezdemo.content_view_provider.class: EzSystems\DemoBundle\Content\ViewProvider

services:
    ezdemo.content_view_provider:
        class: %ezdemo.content_view_provider.class%
        tags:
            - {name: ezpublish.content_view_provider, priority: 30}
Tag attribute nameUsage
priority

An integer giving the priority to the ContentViewProvider in the ViewManager.

The priority range is from -255 to 255

Example

<?php

namespace Acme\DemoBundle\Content;
use eZ\Publish\MVC\View\ContentViewProvider,
    eZ\Publish\API\Repository\Values\Content\Location,
    eZ\Publish\API\Repository\Values\Content\ContentInfo,
    eZ\Publish\MVC\View\ContentView;

class ViewProvider implements ContentViewProvider
{
    /**
     * Returns a ContentView object corresponding to $contentInfo, or void if not applicable
     *
     * @param \eZ\Publish\API\Repository\Values\Content\ContentInfo $contentInfo
     * @param string $viewType
     * @return \eZ\Publish\MVC\View\ContentView|void
     */
    public function getViewForContent( ContentInfo $contentInfo, $viewType )
    {
        // ContentType identifier (formerly "class identifier")
        switch ( $contentInfo->contentType->identifier )
        {
            // For view full, it will load AcmeDemoBundle:full:small_folder.html.twig
            case 'folder':
                return new ContentView( "AcmeDemoBundle:$viewType:small_folder.html.twig" );
                break;
        }
    }
    /**
     * Returns a ContentView object corresponding to $location, or void if not applicable
     *
     * @param \eZ\Publish\API\Repository\Values\Content\Location $location
     * @param string $viewType
     * @return \eZ\Publish\MVC\View\ContentView|void
     */
    public function getViewForLocation( Location $location, $viewType )
    {
        // Let's check location Id
        switch ( $location->id )
        {
            // Special template for home page, passing "foo" variable to the template
            case 2:
                return new ContentView( "AcmeDemoBundle:$viewType:home.html.twig", array( 'foo' => 'bar' ) );
                break;
        }
 
        // Fallback to ContentInfo based rules
        return $this->getViewForContent( $location->getContentInfo(), $viewType );
    }
}

Parameters injection in content views

It is possible to dynamically inject variables in content view templates by listening to the ezpublish.pre_content_view event.

The event listener method receives aneZ\Publish\MVC\Event\PreContentViewEvent object

Example

<?php

namespace Acme\DemoBundle\EventListener;
use eZ\Publish\MVC\Event\PreContentViewEvent;

class PreContentViewListener
{
    public function onPreContentView( PreContentViewEvent $event )
    {
        // Get content view object and inject whatever no need.
        // You may add some custom business logic here.
        $contentView = $event->getContentView();
        $contentView->addParameters(
            array(
                 'foo'          => 'bar',
                 'osTypes'      => array( 'osx', 'linux', 'win' )
            )
        );
    }
}

Render embedded content objects

Rendering an embedded content from a Twig template is pretty straight forward:

{% render "ez_content:viewLocation" with {"locationId": 123, "viewMode": "full"} %}

ESI

Just as for regular Symfony controllers, you can take advantage of ESI and use different cache levels:

{% render "ez_content:viewLocation" with {"locationId": 123, "viewMode": "line"}, {"standalone": true} %}

Asynchronous rendering

Symfony also supports asynchronous content rendering with the help of hinclude.js library.

{% render "ez_content:viewLocation" with {"locationId": 123, "viewMode": "line"}, {"standalone": "js"} %}
hinclude.js needs to be properly included in your layout to work.
